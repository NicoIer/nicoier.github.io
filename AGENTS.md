# AI Agent 写作风格指南

## 适用范围

- 本文件用于指导 AI Agent 按照仓库作者既有文章风格，总结技术文档、源码实现、工具使用记录或问题排查过程。
- 分析样本来自 `source/_posts` 中 `date < 2026-05-01` 的文章，共 37 篇。
- 样本主题主要是 Unity、Shader、图形学、C#、游戏开发工具链、性能优化和少量投资基础知识。
- 2026 年 6 月及之后的投资分析系列未纳入样本，避免引入后期模板化投资报告风格。

## 总体风格画像

作者的文章更像工程笔记和问题复盘，不像正式论文、营销文或完整教程。核心目标是把一个问题、一个 API、一个实现方案或一个踩坑记录讲清楚，方便自己和后来的人快速复用。

整体语言直接、务实、偏口语化。经常先给背景或结论，再解释原理、列步骤、贴代码、说明代价或注意点。文字不追求华丽，也不刻意制造情绪，重点是信息密度和可操作性。

## 语言习惯

### 语气

- 使用中文技术笔记语气，直接说明事实、判断和经验。
- 可以带一点个人判断，例如“比较方便”“有点垃圾”“罪大恶极”“说白了就是”，但不要滥用情绪化表达。
- 不要写成公众号营销腔，不要写“你是否遇到过”“本文将带你深入探索”这类句子。
- 不要过度客套，不要大量使用“我们可以看到”“众所周知”。

### 常用表达

优先使用这些连接方式和判断方式：

- “这里记录一下……”
- “首先……然后……”
- “本质上是……”
- “实际上……”
- “通常……”
- “一般……”
- “可以通过……实现”
- “需要……”
- “导致……”
- “解决方案是……”
- “注意……”
- “说白了就是……”
- “直接……即可”
- “比如……”
- “例如……”
- “也就是说……”
- “所以……”
- “从而……”

### 句子长度

- 以短句和中等长度句子为主。
- 每段通常只承担一个意思。
- 复杂概念可以拆成多段，不要堆成长段。
- 可以直接使用英文技术名词，不必强行翻译。

### 中英混写

- 技术名词保留英文，例如 `Shader`、`RenderTarget`、`CommandBuffer`、`SRP Batcher`、`GPU Instancing`、`AssetBundle`。
- 中文和英文之间可以自然混排，不强求统一空格，但保持可读。
- API、类名、方法名、文件名、命令行使用反引号包裹。

## 结构习惯

### 常见文章结构

优先采用下面几类结构。

#### 问题记录型

```md
# 问题名称

先说结论：……

问题现象是……

本质上是由于……

解决方案是……

代价/注意点：
- ……
- ……
```

#### 技术整理型

```md
# 技术名词或主题

一句话解释它是什么，以及在什么场景下用。

## 背景

为什么需要这个东西，它解决什么问题。

## 使用方法

- 第一步……
- 第二步……
- 第三步……

## 细节说明

解释关键 API、数据结构或运行机制。

## 参考

- 官方文档或相关链接
```

#### 实现方案型

```md
# 功能或方案名

本文记录了……的实现过程。

## 原方案

原先的方案是……

这样做的问题：
- ……
- ……

## 新方案

新的方案是……

## 代码

贴关键代码，不必贴无关工程代码。

## 注意点

- ……
- ……
```

#### 命令速记型

````md
# 操作名称

简单说明用途。

# 步骤一

```shell
command
```

# 步骤二

```shell
command
```
````

## 标题风格

- 标题直接使用技术主题或问题名称。
- 不使用夸张标题。
- 二级标题常见写法：
  - `背景`
  - `使用方法`
  - `示例代码`
  - `细节说明`
  - `代价`
  - `注意点`
  - `参考文章`
  - `源代码`
- 可以使用问题式标题，例如 `手机上AlphaClip为什么会很卡`、`需要序列化什么？`、`如何反序列化？`。

## 内容组织规则

- 如果有明确结论，优先把结论放前面。
- 如果是问题排查，按“现象 -> 原因 -> 解决方案 -> 代价/注意点”的顺序写。
- 如果是技术介绍，按“是什么 -> 为什么需要 -> 怎么用 -> 细节 -> 参考”的顺序写。
- 如果是代码实现，先讲数据结构和关键流程，再贴代码。
- 能用列表就用列表，尤其是步骤、优缺点、注意点。
- 代码块只放关键代码，不需要完整工程。
- 公式、矩阵、链接、图片可以直接插入，不需要过多铺垫。
- 可以用一句非常短的话作为总结，例如“看上去是对的，那就是对的”，但不要为了模仿而强行加金句。

## 总结文档时的输出规范

AI Agent 在总结外部文档或代码时，优先按下面格式生成。

```md
# 文档或主题名称

一句话说明这份文档主要解决什么问题。

## 核心结论

- ……
- ……
- ……

## 背景

说明为什么会有这个问题，或者这个技术出现的原因。

## 关键概念

- `概念A`：……
- `概念B`：……
- `概念C`：……

## 使用方法

- ……
- ……
- ……

## 实现细节

说明关键流程、数据结构、API 调用或渲染/运行时机制。

## 注意点

- ……
- ……

## 参考

- ……
```

如果原始内容很短，可以省略 `背景`、`实现细节`、`参考` 等章节，直接写结论和步骤。

## 摘要压缩原则

- 优先保留可执行信息：命令、参数、API、代码入口、配置项、限制。
- 优先保留因果关系：什么导致什么，为什么这样做。
- 删除泛泛而谈的介绍和重复解释。
- 不要把每个概念都扩写成百科词条。
- 不要编造没有出现在原文中的性能数据、版本要求或限制条件。
- 如果不确定，直接写“不确定”或“原文没有说明”。

## 应该模仿的写法

```md
先说结论：Unity 中的阴影距离实际上是 near plane + shadow distance。

问题主要出在 AlphaClip 的 OverDraw。AlphaClip 会丢弃像素，但这些像素在片元着色器阶段之前已经被处理过了，所以仍然会消耗 GPU 资源。

解决方案是先做一遍 Pre-Z，只写深度。然后在 AlphaClip Pass 中使用 `ZTest Equal`，这样没有通过深度测试的像素就不会进入片元着色器。

代价是会多一次绘制，所以这个方案适合片元开销明显大于顶点开销的场景。
```

## 应该避免的写法

```md
在现代实时渲染领域，AlphaClip 一直是开发者必须面对的重要课题。本文将带你从底层原理到工程实践全面掌握这一技术，并帮助你构建高性能、高质量的移动端渲染方案。
```

这种写法太像教程营销文，不符合作者原有风格。

## 生成前检查清单

- 是否排除了 2026 年 6 月之后投资分析模板的影响。
- 是否保留了工程笔记的直接感。
- 是否先讲结论或问题，再讲原因和方案。
- 是否使用了足够的列表、代码块和技术名词。
- 是否避免了过度文学化、营销化、论文腔。
- 是否没有编造原文不存在的细节。
- 是否能让读者看完后知道下一步怎么做。
