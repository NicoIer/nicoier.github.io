# 伤害计算

这里记录一下当前版本的伤害公式，只处理四个东西：

- 基础伤害
- 固定加减伤
- 增伤倍率
- 减伤倍率

普攻、技能直伤和简单 Buff 先用这套就够了。暴击、护盾、元素抗性、PvP 修正后面单独加阶段，不继续往 `damage_increase` 和 `damage_reduction` 里塞。

{% raw %}
$$
\begin{aligned}
raw\_damage &= \max(0, base\_damage + damage\_offset) \\
increase\_multiplier &= 1 + damage\_increase \\
reduction\_multiplier &= \max(0, 1 - damage\_reduction) \\
final\_damage &= raw\_damage
  \times increase\_multiplier
  \times reduction\_multiplier
\end{aligned}
$$
{% endraw %}

## 字段说明

| 字段 | 说明 |
| --- | --- |
| `base_damage` | 基础伤害，通常来自技能表或攻击力系数 |
| `damage_offset` | 固定加减伤，可以是负数 |
| `damage_increase` | 增伤比例，20% 写成 `0.2`；同一乘区先相加 |
| `damage_reduction` | 减伤比例，30% 写成 `0.3` |
| `raw_damage` | 固定修正后的伤害，最低裁到 0 |
| `final_damage` | 最终结算值，取整规则由服务端统一 |

## 计算顺序

1. 先算基础伤害和固定修正，得到 `raw_damage`。
2. 把同一乘区内的增伤加总，得到 `increase_multiplier`。
3. 把减伤转成 `reduction_multiplier`。
4. 得到 `final_damage`。

顺序固定后，伤害不对时直接看中间值即可。不要让每个技能自己拼最终公式。

## 是否还需要扩展

后面需要补的阶段：

- 暴击，比如 `critical_multiplier`
- 护盾、吸收、免疫，一般放在最终伤害之后、扣血之前
- 元素、护甲、抗性，单独做防御乘区
- PvP / PvE 修正
- 技能增伤、易伤、最终伤害等独立乘区
- 最小伤害和单次伤害上限

重点是每个阶段的输入输出要确定。每个 Buff 都能随便改任意字段，最后肯定没法查。

## 多行 LaTeX 写法

Hexo 里的多行公式用 `aligned`，外面再套一层 raw 标签，否则 Markdown 可能把换行转成 `<br>`。上面的伤害公式就是完整写法。

## 注意点

- 服务端和客户端必须使用同一套取整规则。
- 同步最终伤害，不让客户端重新算一遍；客户端预测也要接受服务端覆盖。
- Buff 修改字段时，最好明确是加到 offset、increase 还是 reduction。
- 不要让显示层直接改伤害字段，飘字只读最终结果。
